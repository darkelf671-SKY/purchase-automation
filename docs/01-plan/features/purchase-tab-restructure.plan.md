# Plan: 구매 조사 탭 UI 재구조화

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 기안 정보 섹션에 12개 항목이 과밀 집중되어 있고, 견적 비교 전에 구매업체·결제방법을 요구하는 순서가 실무 프로세스(품목확인→시장조사→견적비교→업체결정→기안작성)와 역행하여 스크롤 왕복과 입력 혼란 유발 |
| **Solution** | 4개 섹션으로 재분리하여 "① 품목·가격 → ② 시장조사·견적 → ③ 업체·결제 → ④ 기안 작성" 순서로 재배치. VAT를 품목 섹션에 통합, 구매업체·결제방법을 견적 비교 이후로 이동 |
| **Function/UX** | 위→아래 순차 입력이 실무 흐름과 일치. 기안 정보 항목 12개→7개로 감소. 스크롤 왕복 제거 |
| **Core Value** | 실무자가 화면 흐름만 따라가면 문서가 완성되는 직관적 입력 경험 제공 |

## 1. 현황 분석 (AS-IS)

### 1.1 현재 섹션 구조와 빌드 순서

```
_build_ui() 호출 순서:
  0. _build_edit_banner()        — 수정/복사 모드 배너
  1. _build_draft_section()      — ❶ 기안 정보 (row 0~12, 12개 항목)
  2. _build_survey_section()     — ❷ 구매 조사 (VAT + 사이트)
  3. _build_items_section()      — ❸ 품목 목록 (동적 테이블)
  4. _build_quote_compare_section() — ❹ 견적 비교 (견적1/2 + 최저가)
  5. 액션 버튼                    — 초기화 + 생성
```

### 1.2 실무 프로세스와 UI 순서 불일치

| 실무 순서 | 실무 행위 | 현재 UI 위치 | 불일치 |
|-----------|-----------|-------------|--------|
| 1단계 | 구매할 품목 확인 | ❸ 품목 목록 (3번째) | 뒤로 밀림 |
| 2단계 | 시장조사 (가격 비교) | ❷ 구매 조사 (2번째) | 약간 밀림 |
| 3단계 | 견적 수집 (2곳) | ❸+❹ 품목+견적 | - |
| 4단계 | 최저가 업체 결정 | ❹ 견적 비교 (4번째) | - |
| 5단계 | 업체·결제 확정 | ❶ 기안 정보 (1번째) | **완전 역행** |
| 6단계 | 기안서 작성 | ❶ 기안 정보 (1번째) | **완전 역행** |

### 1.3 기안 정보 섹션 과밀화 상세

현재 `_build_draft_section()` 내부 (grid row 0~12):

| Row | 항목 | 실제 성격 | 결정 시점 |
|-----|------|----------|----------|
| 0 | 템플릿 선택/적용/저장 | 기안 작성 | 5~6단계 |
| 1 | 구분선 | - | - |
| 2 | 기안제목 * | 기안 작성 | 5~6단계 |
| 3 | 기안일 | 기안 작성 | 5~6단계 |
| 4 | 부서명 (읽기전용) | 고정값 | - |
| 5 | 내용 * (Text 4행) | 기안 작성 | 5~6단계 |
| 6 | 비고 | 기안 작성 | 5~6단계 |
| 7 | 구분선 | - | - |
| 8 | 포함항목 (계약/수의) | 문서 옵션 | 5~6단계 |
| 9 | **구매업체 *** | **업체 결정** | **4~5단계** |
| 10 | **업체 정보 (자동)** | **업체 결정** | **4~5단계** |
| 11 | **결제방법 *** | **결제 결정** | **4~5단계** |
| 12 | **은행 정보** | **결제 결정** | **4~5단계** |

Row 9~12 (구매업체, 업체정보, 결제방법, 은행정보)는 견적 비교 후에 결정되는 항목이나 기안 정보 최상단에 위치 → **스크롤 왕복 필수**

## 3. 코드 빌드 순서 제약조건

### 3.1 반드시 지켜야 하는 의존 순서

QA 코드 분석 결과, 다음 빌드 순서 제약이 존재합니다:

| 제약 | 이유 | 근거 코드 |
|------|------|-----------|
| **③ < ②** (업체 빌드 < 견적 빌드) | `on_vendor_selected()`가 `_draft_vendor_var`, `_on_draft_vendor_select()` 참조 | line 998-999 |
| **① (VAT) < ① (품목)** | `_add_item_row()` 시 `_vat_radios` 참조 가능성 | `_check_total_mode_vat()` line 890 |

### 3.2 안전한 빌드 순서 결정

```python
def _build_ui(self):
    self._build_edit_banner()           # 0. 배너
    self._build_items_section()         # ① 품목 (VAT 포함)
    self._build_survey_quote_section()  # ② 시장조사 + 견적
    self._build_vendor_payment_section()# ③ 업체·결제 (신규)
    self._build_draft_section()         # ④ 기안 작성 (축소)
    # 기안제목 자동채움 trace
    # 액션 버튼
```

**안전 보장**: ③(업체 위젯 생성) → ②(견적, on_vendor_selected에서 ③ 위젯 참조) 순서이나,
실제로 `on_vendor_selected`는 사용자 이벤트 시 호출되므로 빌드 완료 후에만 실행됨.
따라서 ①→②→③→④ 순서로 빌드해도 런타임에는 문제 없음.

단, `_check_total_mode_vat()`가 `_vat_radios`를 참조하므로 VAT 라디오버튼이 먼저 빌드되어야 함 → VAT를 ① 품목 섹션 상단에 배치하면 해결.
